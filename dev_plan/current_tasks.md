# 当前任务（MVP 分阶段）

说明：这是把 `dev_plan/development_outline.md` 中的 MVP（最小可行产品）拆分成更小且可逐步完成的阶段化任务列表。每完成一项，请把对应的复选框标为已完成，并在该项下追加完成记录（时间、执行者）。

---

## 总览（来自 development_outline.md 的 MVP）
- 手牌系统：购买、持有、打出卡牌
- 建筑放置与生产：建筑放置后周期性产兵
- 单位战斗：简单自动寻路/直线移动并互相攻击
- 双方城堡：血量系统与胜负判定
- 简易商店界面与货币系统

---

## 分阶段任务（优先顺序）

### 阶段 A — 项目骨架与公共基础
- [ ] A.1 项目初始化与目录/场景模板（创建主场景、基础输入、基础 UI 层）
- [ ] A.2 引导性 README 与开发说明（确保 `dev_plan/` 中大纲指向并记录约定）
- [ ] A.3 自动加载（Autoload）基础管理器空壳：`GameManager`, `UIManager`（C# 脚本）

备注：阶段 A 为所有后续工作打基础，优先完成。负责人：复合（助手/开发者）。

---

### 阶段 B — 货币与商店（先实现经济基础，方便测试卡牌流转）
- [ ] B.1 经济系统骨架（`Economy` / `CurrencyManager`）：货币增减 API、持久化基本数据结构
- [ ] B.2 商店 UI 协议（UI 画布占位，能够显示卡牌项与价格）
- [ ] B.3 商店购买流程（消费货币、将卡牌加入手牌）
- [ ] B.4 测试场景：玩家可在商店购买一张示例卡并看到货币变动

---

### 阶段 C — 手牌系统（CardSystem）
- [ ] C.1 卡牌数据结构（Card 数据类 / 资源模板，存放在 `assets/art/cards/` 引用示例）
- [ ] C.2 `CardSystem` 骨架：购买、持有、手牌上限、打出卡牌接口
- [ ] C.3 手牌 UI（显示手牌格、能触发“打出”动作）
- [ ] C.4 将商店购买流程与 `CardSystem` 集成（完成后可从商店取得卡牌并放入手牌）

---

### 阶段 D — 建筑放置与产兵（BuildingSystem）
- [ ] D.1 建筑预制体模板（prefab），含生成点/产兵参数（生产间隔、产出单位类型）
- [ ] D.2 放置逻辑：从手牌打出建筑 → 在战场可放置位置创建建筑实例
- [ ] D.3 生产计时器实现：建筑周期性触发单位生成事件
- [ ] D.4 单位生成接口（与 UnitSystem 协作）

---

### 阶段 E — 单位与战斗（UnitSystem + BattleSystem）
- [ ] E.1 单位预制体骨架：生命值、伤害、移动速度、基础碰撞/碰撞层设置
- [ ] E.2 简易移动/AI：直线前进或朝目标前进（先实现直线移动以保证可见战斗）
- [ ] E.3 攻击与伤害结算（近战/远程基础逻辑）
- [ ] E.4 单位死亡处理与清理
- [ ] E.5 战场协调（生成、碰撞检测、战斗状态）

---

### 阶段 F — 城堡系统与胜负判定
- [ ] F.1 城堡实体（Castle）与血量系统（HP）显示到 UI
- [ ] F.2 单位攻击城堡判定与伤害计算
- [ ] F.3 胜负判定（城堡血量 <= 0 → 结束回合/游戏）
- [ ] F.4 一局流程验证：从购买、放置、产兵、交战到胜负判定的完整可演示流程

---

### 阶段 G — 集成测试与验证（MVP 收尾）
- [ ] G.1 单元测试：关键算法（生产计时、伤害计算）
- [ ] G.2 关卡/场景测试：不同建筑/单位组合的稳定性测试
- [ ] G.3 简易优化/性能检查（必要时）
- [ ] G.4 MVP 演示场景准备（用于内部 Demo）
